正确安全的加密密码

web端开发一定离不开对用户密码的保存，关于如何正确的加密密码网上也有很多值得学习的文章，这篇文章主要是为了为这一块知识进行积累，加深记忆。

不要用自己的法子去加密

有些时候，我们可能打瞌睡的时候突然想到一个加密的野路子，觉得谁也想不到这个野路子，想拿它来加密用户的密码。这种情况一定要避免，即使路子再野，也不安全。因为黑客攻击者已经掌握了哈希函数的参数了，你可以去那种解密网站看看，常见的密码即使使用sha256也可以在很快的速度解密出来，如果你所在的项目是一个很重要的项目，攻击者使用昂贵的设备可以在更快的速度解密出来，这样可能就会造成十分严重的损失，所以，我们不是专业的密码学科学家，还是不要浪费功夫写自己的加密了。

哈希加密

哈希加密函数是一种单向的加密函数，我们可以利用哈希函数将明文转化成密文，但是却不可以将密文利用哈希函数反向转回来。因此正常流程上，当已经储存好了用户的密码的密文在数据库后，在用户登陆时，比较用户登录密码的加密密文和数据库中储存的密文是否相同。

要注意，如果用户输入错了密码，千万不要只提示密码错误。而应该提醒用户名或密码错误。因为如果是攻击者，只提示密码错误，攻击者就可以确定用户名是否是正确的用户名。

破解哈希

如何破解哈希呢？破解哈希通常有两种法子：

  • 字典攻击：字典攻击是使用包含单词、短语、常用密码和其他可能用做密码的字符串的字典文件。对文件中的每个词都进行哈希加密，将这些哈希值和要破解的密码哈希值比较。如果它们相同，这个词就是密码。字典文件是通过大段文本中提取的单词构成，甚至还包括一些数据库中真实的密码。
  • 暴力攻击 : 暴力攻击就是一个一个密码去尝试是否哈希后的结果与数据库中的相同，暴力攻击因为要遍历每一种可能，需要耗费大量的资源，如果密码十分的复杂，暴力攻击破解出密码将会耗费大量的时间，甚至攻击者也无法承担。但是利用暴力攻击一定会找出密码，只不过是时间问题了。

目前没有办法来防止字典攻击或暴力攻击。只能想办法让它们变得低效。如果密码哈希系统设计是安全的，破解哈希的唯一方法就是进行字典攻击或暴力攻击遍历每一个哈希值了。这就使得攻击者攻击的时间代价、成本代价都无法承担。

加盐

因为现在常用的哈希加密函数已经基本被人破解了，如果我们只使用这些哈希函数来加密，和不加密基本没有什么不同。因此我们必须使用一个随机值混合着密码然后利用哈希函数进行加密。这个被加的字符串称之为盐值（salt）。往数据库储存的时候，通常是把 这个salt和加密后的密文一起放入数据库。

salt 随机化了哈希值，让查表法、反向查表法和彩虹表都会失效。因为攻击者无法事先知道盐值，所以他们就没有办法预先计算查询表或彩虹表（除非他把数据库脱裤了）。如果每个用户的密码用不同的盐再进行哈希加密，那么反向查表法攻击也将不能奏效。

错误的加盐使用方法

  • 短盐值
    • 如果选用的盐值的长度太短，黑客就可以根据遍历所有的可能性来破解你的盐值。
  • 盐值复用
    • 如果出现了盐值复用，一般就是因为把盐值硬编码到代码中，这样会导致一种危险的情况，如果有两个用户的密码是相同的，那么通过这个重复的盐值，就会通过哈希函数生成两个相同的密文，如果这种相同密码的情况很多，那么在数据库中就会出现很多相同的密文。攻击者依旧可以使用字典攻击来破解出密码。

为使攻击者无法构造包含所有可能盐值的查询表，盐值必须足够长。例如， SHA256 的输出为 256 位（32 字节），所以该盐值的长度最好也应该是 32 个随机字节。

双重哈希

在以前不了解哈希加密的知识前，我曾以为使用下面的做法来加密会很安全。尝试不同的哈希函数相结合一起使用，以为让数据会更安全。

  • md5(sha1(password))
  • md5(md5(salt) + md5(password))
  • sha1(sha1(password))
  • sha1(md5（password + salt))
  • md5(sha1(md5(md5(password) + sha1(password)) + md5(password)))）

如果创造新的哈希函数，可能会带来风险，构造希函数的组合又会导致函数互通性的问题。它们带来一点的好处和这些比起来微不足道。很显然，最好的办法是，使用标准、经过完整测试的算法。要使用更安全的哈希函数，比如 SHA256 、 SHA512 、 RipeMD 或 WHIRLPOOL 。

如何正确进行哈希加密

加盐哈希

上面已经说到了，通过将密码和随机且唯一的盐值混合一起进行哈希加密是安全的做法。那么我们应该怎么获得这样的盐值呢？

盐值应该使用安全的加密的伪随机数生成器（ Cryptographically Secure Pseudo-Random Number Generator，CSPRNG ）来生成。

CSPRNG 被设计成用于加密安全，这意味着它能提供高度随机、完全不可预测的随机数。我们不希望盐值能够被预测到，所以必须使用 CSPRNG 。

在Java中，CSPRNG的代表便是java.security.SecureRandom

每一个用户的盐值都应该是唯一且不重复的，在每次创建新账号和修改密码时，都应该使用一个随机的盐值。

储存密码的步骤

  • 使用CSPRNG来生成一个足够长的盐值
  • 将盐值混入明文密码，然后使用混合后的字符串使用标准的安全哈希函数来加密，PBKDF2 ，bcrypt ，scrypt
  • 然后将 密文密码和盐值一起存入数据库中。

校验密码的步骤

  • 根据用户的用户名从数据库中校验出对应的盐值和密文密码
  • 根据用户输入的密码，通过对应的哈希函数来生成对应的密文
  • 拿着生成的密文和数据库中储存的密文进行比较，相同则通过，不相同则驳回。

记住一点，加密永远在服务端进行加密，即使你在前端加密过一次了，数据走到服务端也要再加密一次。这是 因为如果你只在前端进行加密，再后端不加密，这就代表什么呢！

当前端把数据往后端传密文密码时，会把这个密文密码存入数据库中，想一下，这样和直接使用明文密码存数据库有什么区别。因为前端传给后端的数据就是要存入数据库的密文密码。攻击者如果脱库了，这就代表攻击者拿到了所有人的密文密码。这个密文后端是直接识别的。那这样和明文是没有任何区别的。

所以，一定要在服务端加密。

慢哈希函数：让密码更难破解

随着计算机硬件的飞速发展，高端显卡的计算速度已经突破每秒亿次了，使用高端硬件的字典攻击，计算速度的提升，让攻击成功的可能性进一步提升了。为了降低使这些攻击的效率，可以使用一个叫做密钥扩展（ key stretching）的技术。

这样做的初衷是为了将哈希函数变得非常慢，即使有一块快速的 GPU 或定制的硬件，字典攻击和暴力攻击也会慢得令人失去耐心。

密钥扩展的实现使用了一种 CPU 密集型哈希函数（ CPU-intensive hash function）。不要试图去创造你自己的迭代哈希加密函数。要使用标准的安全算法，比如 PBKDF2 或 bcrypt 。这类算法采取安全因子或迭代次数作为参数。此值决定哈希函数将会如何缓慢。对于桌面软件或智能手机应用，确定这个参数的最佳方式是在设备上运行很短的性能基准测试，找到使哈希大约花费半秒的值。通过这种方式，程序可以尽可能保证安全而又不影响用户体验。

如果您想在一个 Web 应用使用密钥扩展，虽然密钥扩展也容易让服务端遭受拒绝服务攻击（ DDoS ）。尽管如此，依然建议使用密钥扩展，只不过要设定较低一些的迭代次数。这个次数需要根据自己服务器的计算能力和预计每秒需要处理的认证请求次数来设置。消除拒绝服务的威胁可以通过要求用户每次登陆时输入验证码（ CAPTCHA ）来做到。
