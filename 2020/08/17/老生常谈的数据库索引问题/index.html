老生常谈的select问题

为什么 Select * from table 在Java开发手册中被建议为不要使用呢？

于不久前推出的JAVA开发手册（嵩山版）针对MySQL的建议中有如下一条

位置于 （五）MySQL数据库（四）orm规范 （第一条）

【强制】在表查询中，一律不要使用 * 作为查询的字段列表，需要哪些字段必须明确写明。 说明：1）增加查询分析器解析成本。2）增减字段容易与resultMap配置不一致。3）无用字段增加网络消耗，尤其是text类型的字段。

在这上面，也大概指出了为什么不用select * form table的原因。

  • 首当其冲的就是，这样会无法使用索引的优点（尤其是覆盖索引），查询效率低下。

    在MySQL中的索引有，主键索引，唯一索引，普通索引。正常情况下，建立了普通索引后，在查询数据时，会先通过普通索引定位到相应的主键id上，然后回到主键索引上，这个行为被称为回表操作，利用普通索引查到的主机id，在主键索引中查出对应的数据。

    因此，为了避免那一次回表，我们可以使用覆盖索引，所谓的覆盖索引，就是普通索引已经包含了所有满足查询需要的数据，这就不需要再回到主键索引上查询数据，避免了一次IO操作，提高了查询效率。

  • 增加查询分析器的解析成本，增加无谓的消耗。

    例如数据库是博客网站的数据库，其中肯定会有text这种字段用于储存文章本体。通常一篇博文的长度都在3000字-10000字左右，而大量的文章的博文在用SELECT *数据库需要解析更多的对象、字段、权限、属性等相关内容，数据传输的长度会几何增涨，在 SQL 语句复杂，硬解析较多的情况下，会对数据库造成沉重的负担。

建立索引的几条建议（选自Java开发手册+自身理解）

  • 建组合索引的时候，区分度最高的在最左边。

    如果where a=? and b=?，a列的几乎接近于唯一值，那么只需要单建idx_a索引即可。

    如果存在非等号和等号混合判断条件时，在建索引时，要把等号条件的列前置。如：where c>? and d=? 那么即使c的区分度更高，也必须把d放在索引的最前列，即建立组合索引idx_d_c。

  • 业务上具有唯一特性的字段，即使是组合字段，也必须建成唯一索引。

    即使在应用层做了非常完善的校验控制，只要没有唯一索引，根据墨菲定律，必然有脏数据产生。

  • 如果有order by的场景，请注意利用索引的有序性。order by 最后的字段是组合索引的一部分

    并且放在索引组合顺序的最后，避免出现file_sort的情况，影响查询性能。

    where a=? and b=? order by c, 对应着索引：idx_a_b_c 。

    索引如果存在范围查询，那么索引有序性无法利用，如：WHERE a>10 ORDER BY b; 索引idx_a_b无法排序。

  • 利用延迟关联或者子查询优化超多分页场景

    SELECT  各种字段
FROM `table_name`
WHERE 各种条件
LIMIT 0,10;
PS: 随着limit后面的数值变得越来越大，这条语句的执行效率会变得的很差，因为offset+N行，是先查询了offset+N行，然后再过滤掉前offset行。因此当offset变得很大，需要查询很多行，执行效率低下是显而易见。
    SELECT  各种字段
FROM `table_name` as t1,
(
SELECT id
FROM `table_name`
WHERE 各种条件
LIMIT 5000000,10;
) as t2 
where t1.id = t2.id

PS:上面的子查询优化就是利用了普通索引，在普通索引中，取出符合条件的N行，只需要将这N行进行回表到主键索引上查询相应数据。避免了上一个例子中offset+N行全部回表的操作。

避免索引的一些误解（都是错误的！）

  • 抵制唯一索引。认为唯一索引可以在应用层通过严格的校验方式解决（业务上具有唯一特性的字段，必须建成唯一索引）。
  • 不管表数据量多少，都建立索引（数据量小的表不需要建立索引，建立会增加额外的索引开销）。
  • 建立大量的索引（数据变更需要维护索引，意味着索引越多维护成本越高）。
  • 对数据重复且分布平均的字段建立索引（例如性别字段，只有男女，不适合建立索引）。
  • 频繁更新的列也建立索引（频繁更新的列不要建立索引，因为会影响插入或更新的效率）。
