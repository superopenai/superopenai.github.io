Redis -过期策略及内存淘汰机制

Redis的用途十分广泛，作为一款高性能的内存数据库，经常被用于数据库缓存，把经常访问的热点数据储存于内存之中，这样省去了数据库频繁的与磁盘进行I/O。但是内存空间是有限的，缓存也不能无限占用内存空间，如果热点数据凉凉了呢？如何清理这些过期的数据呢？

Redis 给我们解决的答案：

  • 过期策略
  • 内存淘汰机制

过期策略

结论：Redis 使用的是 定期删除 和 惰性删除 配合的过期策略。

  • 定期删除： Redis 会每隔100ms 随机的检测一些设置了过期时间的Key，检测这些Key是否已经过期。如果已经过期，就将其删除。（不能进行全盘扫描Key的状态，因为那样将会占用大量的系统资源）
  • 惰性删除： 因为只靠 定期删除 是肯定无法去除所有的过期Key的，因此需要配合惰性删除，所谓惰性删除，意如其名，只有在访问Key要取Value的时候，再检测这个Key的过期状态，若已经过期，将其删除，并且返回空，否则便返回访问的Key的Value。

不使用定时删除的原因 : 因为如果要使用定时删除，需要为每一个Key 额外的添加一个定时器来负责监视key，当这个key过期就自动删除，虽然内存及时释放，但是十分消耗CPU资源，因此一般不推荐采用这一策略。

内存淘汰机制

所谓内存淘汰，就是Redis设置了一个可以使用的内存空间阈值，一旦占用的空间大小达到了阈值，便会执行设置的淘汰机制，删除一些数据以清理出空间储存新数据。

# maxmemory  配置内存阈值
# maxmemory-policy noeviction 选择淘汰机制

maxmemory 0

当把内存阈值设置为 0 的时候，或默认无限大，在32位系统之下，会隐性的设置成为3GB大小。

  • noeviction ： 当Redis占用空间已经达到极限，这时如果一条命令想要占用更多的空间，Redis将会返回错误。
  • allkeys-lru ： 这时Redis将会删除最近最不常用的Key，以为新数据腾出空间
  • volatile-lru ：仅在设置了过期时间的Key中，选择最近最不常用的Key来删除，为新数据腾出空间。
  • allkeys-random ：随机的删除一些Key，为新数据腾出空间。
  • volatile-random ：在设置了过期时间的Key中，随机的删除一些Key，为新数据腾出空间。
  • volatile-ttl ：删除所有设置了过期时间的Key，然后尝试删除具有较短TTL（生存周期）的Key，新数据腾出空间。

其中 volatile-lru,volatile-random,volatile-ttl 如果条件不符合，其结果将会和 noeviction 一样。

回收(删除)过程如何工作

  • 首先，客户端运行一条命令，这时需要添加新数据
  • Redis检查是否达到了内存阈值，如果达到了，根据设置的淘汰机制，清理空间以便添加新数据。
  • 执行新命令。

这样会造成一种现象，那就是Redis已经到达了内存阈值，然后为了新数据，根据机制释放了一些空间，然后被新数据又占满。再往后为新数据在释放空间，然后又被占满。这时如果一次申请大量空间，大大的超过了内存阈值，就会出现意想不到的结果。

Redis回收使用的算法

Redis使用的是LRU算法，但只是近似的实现，不是LRU算法的精确实现。

这也就意味着我们选择删除的Key并不是最优选择（整体上最近最少访问的）。相应的，Redis实现的是一种近似的LRU算法，通过选取少量的样本，在样本当中删除最优的Key（最近最少访问）。

自Redis 3.0 以后，Redis改进了近似的LRU算法，提高了性能，使得选择出来的Key更加近似于真实的Key。 Redis LRU算法的重要之处在于，可以通过更改每次逐出检查的样本数来调整算法的精度。 样本数越多，精度越高。

maxmemory-samples 10

上面的图像展示了 LRU算法和Redis实现的LRU算法，已经不同版本，不同样本数之间的区别。

  • 浅灰色带是被删除的对象。

  • 灰色地带是未被删除的对象。

  • 绿色带是添加的对象。

与Redis2.8相比，Redis3.0使用5个样例做得更好，但是大多数最近访问的对象仍然保留在Redis2.8中。在Redis3.0中样本量为10的情况下，近似结果与Redis3.0的理论性能非常接近。

但是，将样本大小提高到10，要牺牲一些额外的CPU使用率，以接近真实的LRU性能。 可以根据实际情况，决定是否要牺牲一些CPU使用率，来提供更高的命中率。
